1. About spnd.

spnd. is a privacy-first personal expense tracker for Android, designed for users in India. The app automatically detects bank transaction SMS messages and converts them into categorized expense records — giving you a clear picture of your spending without any manual effort.

spnd. is built on a core principle: your financial data belongs to you and only you. The app operates entirely offline. There are no user accounts, no registration, no cloud sync, and no data collection of any kind.

2. SMS Access — Core Functionality Restricted Permission

spnd. requests READ_SMS and RECEIVE_SMS permissions to provide its core functionality: automatic transaction tracking from bank SMS alerts.

2.1 Why SMS Access Is Required

In India, banks send transaction alerts via SMS for every debit and credit. These SMS messages are the only real-time, programmatic source of transaction data available to consumers. Indian banks do not provide open APIs, webhooks, or any other mechanism for personal finance apps to access transaction data. SMS reading is the only technically viable method to automatically track expenses.

Without SMS access, users would need to manually enter every transaction — making the app significantly less useful and defeating its core purpose.

2.2 How SMS Processing Works

Bank sends SMS → spnd. receives SMS
↓
Is it from a known bank sender? → No → Ignored completely
↓ Yes
Is it a transaction alert (debit/credit)? → No → Discarded
↓ Yes

↓
No data is ever transmitted. Zero network calls.

2.3 What We Read vs. What We Ignore

✓ What We Read

  • Bank debit alerts (e.g., "Rs 500 debited from A/c XX1234")
  • Bank credit alerts (e.g., "Rs 10,000 credited to A/c XX5678")
  • UPI sent/received notifications (e.g., "Sent Rs 200 to Rahul", "Received Rs 500 from Priya")
  • Transfer alerts (NEFT, IMPS, RTGS between accounts)
  • Messages from verified Indian bank senders only (e.g., VM-ICICIB, AD-SBIINB, BZ-AXISBK)

✗ What We NEVER Read

  • OTP / verification codes — detected and skipped immediately
  • Personal messages from contacts
  • Promotional or marketing SMS
  • Non-banking SMS of any kind
  • WhatsApp, Telegram, or other app messages

SMS permission is entirely optional. Users can deny SMS permission during onboarding and use spnd. in manual mode — adding transactions by hand. SMS reading can also be disabled at any time from within the app (Settings → SMS Import → toggle off) or from Android system settings (Settings → Apps → spnd. → Permissions → SMS → Deny).

2.5 Background SMS Processing

When granted RECEIVE_SMS permission, spnd. uses Android's WorkManager to process incoming bank SMS in the background. This ensures transactions are captured even when the app is not open.

  • Background processing follows the exact same rules — only bank transaction SMS are processed
  • Processing takes less than 1 second per SMS
  • No persistent background service runs — the app wakes only when a new SMS arrives
  • Users can revoke this permission at any time

3. Contacts Access Optional Permission

spnd. requests READ_CONTACTS permission for one purpose only: to allow users to pick a contact when creating merchant rules in Settings. This helps users map a phone number to a merchant name for better transaction categorization.

  • Contacts are accessed only when the user explicitly taps "Pick from contacts" in the Merchant Rules screen
  • Only the selected contact's name and phone number are used — to populate the rule form
  • No contact data is stored, indexed, or transmitted
  • This permission is optional — users can type merchant details manually instead
  • The contact picker is the standard Android system contact picker

spnd. uses a zero-server architecture. There is no backend, no API, no cloud database, and no remote server of any kind. All data resides exclusively on the user's device.

7. Data Sharing

spnd. does not share any user data with:

  • The developer or any affiliated company
  • Third-party analytics providers (Google Analytics, Firebase Analytics, Mixpanel, etc.)
  • Advertising networks or data brokers
  • Other apps installed on the device

Zero third-party SDKs for tracking. spnd. does not integrate any analytics, advertising, or tracking SDK. The INTERNET permission is used solely for optional Google Drive backup — initiated by the user, stored in the user's own Google Drive account, encrypted with a user-chosen password.

8. Google Drive Backup (Optional)

Users may optionally back up their data to their own Google Drive account.

  • Backup is user-initiated — never automatic
  • Data is encrypted with AES-256 using a password the user sets
  • The backup file is stored in the user's own Google Drive — the developer has no access
  • Google Sign-In is used only to authenticate with Google Drive API — no user profile data is read or stored
  • Users can delete the backup file from their Google Drive at any time

10. User Consent Flow

spnd. follows a transparent, step-by-step consent process during onboarding:

  1. SMS Permission Screen — clearly explains why SMS access is needed, with an option to skip
  2. Android System Dialog — the standard Android runtime permission dialog is shown; the user must actively tap "Allow"
  3. Historical Import Choice — user chooses how far back to import SMS (1 month, 3 months, or custom date)
  4. Contacts Permission — requested separately with clear explanation; can be skipped

At no point is any permission enabled silently or without the user's explicit action.

11. Data Retention

  • Transaction data — retained on device for as long as the app is installed. Permanently deleted on uninstall.
  • Original SMS text — stored locally as a reference alongside parsed transaction data. Never transmitted. Deleted on uninstall.
  • Settings & preferences — stored locally in EncryptedSharedPreferences. Deleted on uninstall.
  • Google Drive backup — stored in the user's own Google Drive until the user deletes it. The developer has no access to this file.
  • No server-side retention — because there are no servers, there is no data to retain outside the device.
